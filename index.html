Project: Automation Testing Framework for EFT Switch Application:

DOMAIN : FinTech | Role: QA Automation Engineer | Technologies: Selenium WebDriver, TestNG, Java, Maven, Git, Apache POI
DESCRIPTION : Developed and implemented an automation testing framework for an EFT (Electronic Funds Transfer) Switch application UI to support regression testing. The framework was designed with a hybrid approach, integrating data-driven testing capabilities for improved flexibility and reusability.

KEY RESPONSIBILITIES :

  • Designed and set up a custom hybrid automation framework from scratch using Selenium WebDriver, TestNG, and Maven.
  • Test cases are implemented and group attribute added using testNG annotation to group the test-cases under regression.
    Implemented regression test suite and run all the test cases under the group among all the test classes.
  • Developed custom TestNG listeners for:
    • Screenshot capture on test failure.
    • Retry logic for flaky test cases using RetryAnalyzer.
    • Enhanced test result reporting with custom XML and POM plugin configurations.
  • Integrated Excel-based DataProvider using TestNG to read test data dynamically. Debugged and fixed issues related to incomplete row reads in Excel files.
  • Utilized Git for version control and collaborated via GitHub on feature branches and merges.
  • Captured and documented test failures with logs and screenshots, aiding in quicker debugging and resolution.
  • Followed Agile practices, participating in regular sprint reviews and stand-ups, while managing tasks in Jira.

The following testing procedures have been conducted at various releases of the EFT switch software product.

Function Testing :

As a part of the UI function testing, the configration for different modules are done and verifies that the configuration data is properly stored in the database and reflects in the UI. The following functions are tested, during each release as regression test -

  • User configuration : configure different users and check the login using the new user credentials.
  • System Configurations : Fields(Host Name, Host IP, SSL Certificate(key store file) location and access password)
  • TCP interface module configuration : configure the input side and output side modules.
  • ISO message translation interface module configuration : configure the input side and output side modules(sourceNode and sinkNode)
  • Configuring source group, destination group, card group and the routing group : Configure.
  • Configure BIN in the card group : The same BIN which we are suppose to send through the ISO message via source simulator.
  • Configure route : Configre a route from Card Group -> Routing Group -> Destination Group -> Output ISO message translation module
  • Testing the commands from UI to the back-end application, the following commands are tested -
    • logger on/off/clear,
    • TRACE on/off/clear,
    • interfaces connect/disconnect,
    • echo,
    • key exchange

Performance Testing :

The performance testing is done using the jmeter. The (0200) message formate is created in the jmeter test plan using the tcp-sampler. Then the load test is simulated by sending large number of messages with a fixed ramp up time. The goal was to check the maximum possible TPS(transactions per second) that can be performed without failure by the EFT switch application. If we send 100 messages with a ramp up time of 1second, it means the system is fired with an input TPS rate of 100. The response messages from the system was recorded using the listerners in the Jmeter.

WireShark TCP Packet Analysing :

Data duplication analysed while the transaction processing at the EFT Swich core application during the performance testing at higher TPS. Then the same test repeated and captured the input data packets and out put data packets and analysis done. Logic modification suggested as a corrective action.
